1) API Contract Hardening (Projects)

 Make key immutable

Ensure PUT /:id cannot change key (even if client sends it).

Add validation: reject update payload containing key with 400 BAD_REQUEST + stable error code.

 Clarify and enforce owner semantics

Treat owner as immutable after creation (Phase 1).

Validate owner format (e.g., must be a non-empty string; if it’s a Mongo ObjectId in your system, validate that).

 Visibility rules (document + enforce minimal RBAC)

Implement Phase 1 authorization rules:

private: only owner can read/update/delete

public: any authenticated user can read, only owner can update/delete

Apply these checks for:

GET /:id

PUT /:id

DELETE /:id

(and Milestones endpoints too—see section C)

{
  "error": "PROJECT_NOT_FOUND",

B) Project Service TODOs (Milestones)

Your Project Service contract includes Milestones endpoints (list/create/update/etc.).

1) Milestone Ownership & Project State Rules

 Before any milestone operation, validate:

project exists

project is active (isActive=true)

 Enforce access rules based on project visibility:

If project is private → only owner can list/create/update milestones

If project is public → any authenticated user can list, only owner can create/update/delete (Phase 1 choice; document it)

2) Milestone Validation

 Validate milestone request body:

title required, min length 2

startDate and dueDate are optional but must be valid ISO dates if present

if both dates present: startDate <= dueDate

 Ensure milestone is always tied to projectId from the URL (ignore body-provided projectId if any)

3) Milestone Query Improvements (Phase 1)

 Add optional query filters:

isCompleted=true|false

from=<date>, to=<date> (filter dueDate range)

 Add sorting default: dueDate asc, createdAt desc
